# Кастомные правила и типы данных

## Типы данных

Числовые ID — сквозной контракт между go-enum (`internal/models/data_type.go`),
YAML-файлами, override-заголовком и API:

| ID | Тип | Источник правил |
|---|---|---|
| 1 | CREDENTIALS | файловые |
| 2 | API_KEYS | файловые |
| 3 | ACCESS_TOKENS | файловые |
| 4 | IP_ADDRESSES | файловые |
| 5 | PERSONAL_DATA | файловые |
| 6 | CUSTOM | только правила, созданные через API |

Чтобы кастомные правила работали, тип данных 6 (`custom`) должен быть в включённых
типах настроек. В поставляемом дефолте он присутствует; включать явно нужно, только
если вы сузили `data_types` и убрали его.

## Сценарии управления правилами (`internal/usecases/rules`)

Сценарии за management API, по пакету на сценарий (`create`, `update`, `delete`,
`setenabled`, `get`, `list`) по command-паттерну (`contract.go`/`usecase.go`/`handle.go`),
собранные координатором `rules`, который отдаёт каждый как свой `CommandHandler`. Каждая
мутация триггерит reloader после записи, поэтому пересборки реестра идут по единому пути
merge-and-swap. Общие блоки — в подпакетах: `errors` (sentinel'ы + `ValidationError`),
`validation` (путь компиляции `Validate`), `builtins` (неизменяемый индекс встроенных).

- `create`/`update`/`delete`: валидация (`ruleIDRe` = `^[a-z0-9_.-]{1,128}$`, известный
  DataType 1–6, `CompileRule` на временном реестре) → запись → reloader.Reload.
  Сценарии независимы и без состояния (нет межмутационного мьютекса); детекция
  конфликтов опирается на проверку существования плюс хранилище. Sentinel-ошибки
  `ErrNotFound`, `ErrBuiltin`, `ErrAlreadyExists` + обёртка `ValidationError`
  маппятся на HTTP-статусы в слое API.
- `setenabled(id, enabled)`: единственная мутация, применимая и к встроенным правилам —
  пишет сохраняемый disabled-набор (`SetRuleDisabled`) и перезагружает. `enabled` —
  вычисляемое состояние (ID ∈ disabled-набор), а не поле `rule.Rule`; `delete`
  best-effort снимает флаг, чтобы пересоздание ID не воскрешало его отключённым.
- `get`/`list`: сценарии чтения, резолвящие источник каждого правила (builtin/custom) и
  состояние enabled, возвращают готовые к ответу представления, чтобы HTTP-хендлеры
  оставались тонкими.

## Добавление встроенного правила (чеклист)

1. Добавьте в `configs/guardrails_regex_rules.yaml` под нужную группу типа данных
   (или перегенерируйте gitleaks-файл, а не правьте его).
2. Добавьте корпус кейсов в `tests/rules/` (позитивные + негативные).
3. Помните, что префикс `(?m)` добавляется автоматически; якорьте `\b`-границами, а не
   `^$`, если не нужна построчная семантика.
4. Значение плейсхолдера становится `<VALUE_N>` — держите его коротким, `[A-Z_]`.
