# Настройки, модель доверия и логирование

## Семантика резолюции настроек

Три слоя, от статического к пофазовому:

1. **Env-дефолты** (`GUARDRAILS_ENABLED`, `GUARDRAILS_MODE`, `GUARDRAILS_DATA_TYPES`)
   парсятся на старте (`settings.ParseDataTypes` / `models.ParseGuardrailsMode` —
   плохие значения паникуют на старте) и становятся начальным in-process-кэшем.
2. **Хранилище** (`settings.Service`): `Load()` на старте читает хранилище; если
   ничего не было сохранено — **засевает хранилище env-дефолтами** (чтобы API всегда
   читал/писал один источник истины). Далее `PUT /v1/settings` → `Update()` (запись +
   атомарная замена кэша), а тикер `RunRefresh` перечитывает хранилище, чтобы другие
   реплики сходились. Важное следствие: **менять env-переменные после первого старта
   бесполезно**, пока в хранилище лежит сохранённое значение (кроме бэкенда in_memory,
   который забывает при рестарте).
3. **Override-заголовок запроса** — `settings.Effective(global, headerValue)`, чистая
   функция (`internal/service/settings/override.go`):
   - пустой заголовок → применяются глобальные настройки;
   - `none` → guardrails пропускаются для запроса;
   - CSV из чисел/имён → **пересечение** с глобальными типами данных (только сужение;
     нельзя включить то, что глобально выключено);
   - пустое пересечение → отключено для запроса;
   - любой неразбираемый токен → весь заголовок игнорируется, действует полная
     глобальная защита (fail toward protection).

**Семантика режима** (`mode` в документе настроек, `enforce`/`detect`): только
глобальная — override-заголовок может сузить типы данных или пропустить запрос, но
никогда не переключает режим. В `detect` (shadow) пайплайн сканирует, увеличивает
метрики срабатываний и `requests_masked_total{mode="detect"}`, пишет аудит-записи
(включите `GUARDRAILS_AUDIT_ENABLED` — это канал пофазовой оценки), но тело запроса не
изменяется, masking state не сохраняется и фаза ответа не демаскирует. Настройки,
сохранённые до появления поля (и любое невалидное сохранённое значение), нормализуются
к `enforce` — отсутствующая конфигурация склоняется к защите. `PUT /v1/settings`
заменяет весь документ: пропуск `mode` сбрасывает его в `enforce`.

## Модель доверия к override-заголовку

Заголовок — **доверенный ввод шлюза**, не клиента. Envoy ОБЯЗАН вырезать его из
входящих клиентских запросов:

```yaml
request_headers_to_remove: [x-guardrails-data-types]
```

Семантика «только сужение» ограничивает урон от подделанного заголовка до «меньше
проверок на одном запросе» — это всё ещё ослабление, поэтому фичу можно полностью
отключить (`GUARDRAILS_OVERRIDE_HEADER=""`).

## Доверие к `x-request-id`

`x-request-id` — ключ masking state, а в топологиях с общим хранилищем и раздельными
запросом/ответом ещё и драйвер межрепличного fallback демаскирования. Он **обязан**
генерироваться шлюзом: никогда не настраивайте Envoy на сохранение
клиент-предоставленного значения, иначе клиент сможет подделать живой request id
жертвы и демаскировать её оригиналы в свой ответ. Зафиксируйте поведение явно на
edge-прокси (Envoy HttpConnectionManager):

```yaml
generate_request_id: true
preserve_external_request_id: false
```

`GUARDRAILS_STATE_KEY_SALT` это **не** заменяет: сервис считает `HMAC(salt, id)` за
вызывающую сторону, поэтому клиент, контролирующий `x-request-id`, полностью обходит
соль. См. [storage/](../storage/) и [operations/](../operations/).

## Логирование

`internal/logging`: `Setup(level, format)` ставит default для slog (JSON- или
text-хендлер, `AddSource: true`). Хелперы повторяют slog, но `Error`/`Fatal` берут
ошибку позиционно и терпимы к nil. Поля лога на область запроса накапливаются в
`requestProcessor.opts` (`AddLogOpt`/`LogOpts`) — path, request_id, model, sse — и
дописываются в местах вызова.
