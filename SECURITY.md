# Политика безопасности

## Сообщение об уязвимости

Пожалуйста, сообщайте о предполагаемых уязвимостях приватно через GitHub Security
Advisories («Report a vulnerability» на вкладке Security репозитория). Не открывайте
публичные issues по проблемам безопасности.

Мы стремимся подтвердить получение отчёта в течение 5 рабочих дней.

## Заметки по безопасности при развёртывании

Сервис по своей природе обрабатывает и временно хранит **чувствительные данные**
(исходные значения, которые он маскирует). При развёртывании:

- **Хранилище masking state**: с бэкендом `redis` или `postgres` исходные чувствительные
  значения покидают память процесса и хранятся (кратко, с TTL) во внешнем хранилище.
  Ограничьте сетевой доступ и доступ по учётным данным к нему; для одиночной реплики
  предпочтителен `in_memory`. Опционально включите шифрование на месте (AES-256-GCM):
  `GUARDRAILS_STORE_ENCRYPTION_ENABLED=true` плюс `GUARDRAILS_STORE_ENCRYPTION_KEY`
  (base64 32-байтный ключ, например `openssl rand -base64 32`) — доставляйте ключ через
  секрет-менеджер, никогда не коммитьте. Семантика — в [docs/storage/](docs/storage/).
- **Override-заголовок**: Envoy ОБЯЗАН вырезать override-заголовок guardrails (по
  умолчанию `x-guardrails-data-types`) из клиентских запросов (`request_headers_to_remove`).
  Заголовок — доверенный ввод шлюза и может сужать (никогда не расширять) проверки.
- **Config API**: config API (:9080) неаутентифицирован. Защищайте на сетевом уровне —
  держите только внутри кластера и никогда не выставляйте через публичный ingress.
- **Логи**: сервис не логирует значения полезной нагрузки (маскированные/немаскированные)
  на уровне info и не имеет debug-путей, печатающих секреты. В любом случае избегайте
  уровня лога `debug` в проде.
